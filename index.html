Раздел 1 · Постановка задачи

Актуальность

Пересдачи в МИРЭА сегодня организуются вручную — бумажные заявления, устные договорённости и фрагментированный обмен информацией.

  • Студент не знает о доступных слотах и не может самостоятельно записаться
  • Преподаватель не имеет единого инструмента для указания своей доступности
  • Учебный отдел тратит часы на согласование между всеми участниками
  • Отсутствует автоматическая проверка конфликтов в расписании
  • Нет единого журнала аудита — невозможно отследить историю действий
  • Бумажные заявления теряются, согласования срываются по человеческому фактору

ЕДИНЫЙ ВХОД

Один поток — SSO с 2FA

Студент авторизуется институтским логином через sso.mirea.ru. Если в Keycloak включена 2FA — после пароля запрашивается одноразовый код.

  • OpenID Connect · PKCE без client_secret
  • code_verifier (64 байта) → code_challenge = SHA-256
  • POST /auth/mirea/login → /auth/mirea/otp при 2FA
  • После входа — gRPC GetMeInfo на attendance.mirea.ru за ФИО

СПОСОБ 2

Email + пароль, отдельно от МИРЭА

Резервный способ для тестовых аккаунтов и служебных пользователей. Не зависит от sso.mirea.ru и работает даже когда SSO недоступен.

  • Пароль хранится как bcrypt-хэш
  • POST /auth/login → тот же JWT, что и SSO
  • Тестовые аккаунты: student01..15@mirea.ru, teacher@mirea.ru, admin@mirea.ru
  • RolesGuard работает одинаково — права не зависят от способа входа

ALREADY_ENROLLED

Нельзя записаться дважды на ту же пересдачу

Внизу карточки — статусная подпись «Вы уже записаны на пересдачу». Сервер вернёт ошибку ALREADY_ENROLLED, кнопка «Записаться» неактивна.

  • Проверка EnrollmentsService.enroll()
  • Коды ошибок: NO_DEBT · ALREADY_ENROLLED · SLOT_FULL · TOO_LATE · NO_RIGHT
  • Защита от дублей даже при двойном клике
  • AuditLog не пишется — действие отклонено до создания записи

Раздел 6 · Технические аспекты

Четыре интеграции с инфраструктурой МИРЭА

SSO

sso.mirea.ru — Keycloak

OpenID Connect PKCE · code_verifier (64 байта) → code_challenge (SHA-256) → авторизация → обмен кода на token. Поддержка 2FA.

gRPC

attendance.mirea.ru — Pulse

gRPC GetMeInfo → ФИО, ID преподавателя, ID группы. Один запрос на сессию, ответ кэшируется.

HTML

lk.mirea.ru — Bitrix ЛК

Эмуляция браузерной сессии, 7-шаговый маршрут редиректов. HTML-парсинг таблицы долгов, SimpleCookieJar, selectStudentProfile.

iCal

schedule-of.mirea.ru

REST API + iCal-фид по преподавателю. Синхронизация раз в 24 ч, проверка VEVENT-конфликтов при создании слота → 400 Bad Request с описанием.

Раздел 6 · Технические аспекты

Безопасность — JWT, Guards, аудит

  • JWT-токен { sub, email, role, iat, exp } — срок действия 15 минут
  • passport-jwt: автоматическая верификация подписи в каждом запросе
  • RolesGuard + @Roles(): декларативное ограничение по роли
  • bcrypt: хэширование паролей для локальных аккаунтов
  • PKCE (Keycloak): авторизация без client_secret, защита от CSRF
  • Axios interceptor: auto-logout фронтенда при 401
  • AuditLog: каждое значимое действие фиксируется
  • Soft delete: пользователи не удаляются физически (deletedAt)
  • class-validator: валидация на всех DTO
Многоуровневая защитаJWT · PKCE · bcrypt · audit
